# QuantMetrics Data Lifecycle System

## Purpose

This document defines how QuantMetrics Suite stores, reduces, archives, and deletes data generated by backtests, analytics runs, experiments, and research iterations.

The goal is:

> Preserve decision evidence. Remove operational noise.

This is not a simple cleanup script.
This is a controlled data lifecycle system for research-grade trading infrastructure.

---

# 1. System Context

QuantMetrics Suite is structured as a coordinated monorepo:

```text
Market Data
→ quantbuild
→ quantlog
→ quantmetrics_os
→ quantanalytics
→ quantresearch
```

Canonical module roles:

| Module             | Responsibility                                            |
| ------------------ | --------------------------------------------------------- |
| `quantbuild/`      | decision logic, strategy, risk guards, backtest/live path |
| `quantbridge/`     | broker adapters and order execution                       |
| `quantlog/`        | event schema, validation, JSONL event logging             |
| `quantmetrics_os/` | run bundling, artifact orchestration, comparisons         |
| `quantanalytics/`  | read-only analytics, reports, guard/funnel analysis       |
| `quantresearch/`   | research registry, experiment dossiers, decisions         |

The suite root is:

```text
C:\Users\Gebruiker\quantmetrics-suite
```

No runtime paths should use old names such as:

```text
quantbuildv1
quantbridgev1
quantlogv1
quantanalyticsv1
```

Versioning belongs in git, schemas, tags, and artifact metadata — not in active folder names.

---

# 2. Core Principle

Not all data has equal value.

QuantMetrics must distinguish between:

1. Runtime noise
2. Debug evidence
3. Research evidence
4. Institutional evidence

Only categories 3 and 4 deserve long-term retention.

---

# 3. Canonical Data Locations

## 3.1 QuantBuild runtime data

```text
quantbuild/logs/
quantbuild/data/quantlog_events/
quantbuild/data/market_cache/
```

### Policy

| Path                               | Policy                                       |
| ---------------------------------- | -------------------------------------------- |
| `quantbuild/logs/`                 | temporary; can be pruned                     |
| `quantbuild/data/quantlog_events/` | source runtime events; reduce after bundling |
| `quantbuild/data/market_cache/`    | never auto-delete                            |

Market cache is expensive and reusable.
It must not be deleted by lifecycle automation.

---

## 3.2 QuantOS run artifacts

Authoritative run artifacts live here:

```text
quantmetrics_os/runs/<experiment>/<role>/
```

Example:

```text
quantmetrics_os/runs/EXP-2025-baseline/single/
```

This folder is the canonical evidence folder for a run.

It should contain:

```text
manifest.json
status.json
config_snapshot.yaml
resolved_config.yaml
quantlog_events.jsonl
metrics_summary.json
results_summary.md
decision.md
analytics/
audit_pack.zip
```

If a run is bundled into QuantOS, this folder becomes more important than loose outputs elsewhere.

---

## 3.3 QuantAnalytics loose output

```text
quantanalytics/output_rapport/
```

This folder may contain repeated or intermediate analytics exports.

### Policy

Loose analytics output is not authoritative unless copied into:

```text
quantmetrics_os/runs/<experiment>/<role>/analytics/
```

After bundling, loose analytics output can be pruned or archived.

---

## 3.4 Mentor packs

Mentor packs live under:

```text
docs/
```

Current validated packs:

```text
docs/v1_1_context_pack_qb_run_20260425T042136Z_dbd1b0cc.zip
docs/v1_1_audit_pack_qb_run_20260425T042136Z_dbd1b0cc.zip
```

These are high-value evidence packs and must not be automatically deleted.

# 4. Data Classification

## 4.1 Ephemeral Data

Temporary runtime artifacts.

Examples:

```text
temporary logs
partial JSON dumps
debug traces
intermediate analytics exports
failed scratch outputs
```

Retention:

```text
0–7 days
```

Default action:

```text
delete
```

---

## 4.2 Diagnostic Data

Useful for debugging but not necessarily research-worthy.

Examples:

```text
full raw event streams
verbose logs
failed run folders
trace-level output
unbundled analytics exports
```

Retention:

```text
7–14 days
```

Default action:

```text
compress or delete after review
```

---

## 4.3 Research Evidence

Evidence used to compare, reject, or promote strategy changes.

Examples:

```text
config_snapshot.yaml
resolved_config.yaml
metrics_summary.json
results_summary.md
decision.md
THROUGHPUT_COMPARE.json
THROUGHPUT_COMPARE.md
guard_attribution.json
EDGE_REPORT.md
PROMOTION_DECISION.md
```

Retention:

```text
30–90 days for candidates
permanent for promoted runs
```

Default action:

```text
keep reduced artifact set
```

---

## 4.4 Institutional Evidence

High-value reference evidence.

Examples:

```text
baseline runs
promoted candidates
audit packs
context packs
production validation runs
cross-period comparisons
```

Retention:

```text
permanent
```

Default action:

```text
never auto-delete
```

---

# 5. Run Lifecycle State Model

Every run folder under `quantmetrics_os/runs/` must contain:

```text
status.json
```

Example:

```json
{
  "run_id": "qb_run_20260425T042136Z_dbd1b0cc",
  "experiment": "EXP-2025-baseline",
  "role": "single",
  "status": "promoted",
  "created_at_utc": "2026-04-25T04:21:36Z",
  "retention_class": "institutional",
  "delete_allowed": false,
  "archive_allowed": true,
  "reason": "validated baseline reference run"
}
```

Allowed statuses:

```text
scratch
candidate
promoted
rejected
archived
protected
```

---

# 6. Status Definitions

## scratch

Exploratory run with no research value yet.

Examples:

```text
quick local tests
broken configs
temporary parameter experiments
```

Policy:

```text
delete after 7 days
```

---

## candidate

Run with potentially useful findings.

Examples:

```text
strong PF but low sample
new guard variant
new session/regime filter
throughput improvement candidate
```

Policy:

```text
keep 30–90 days
compress raw logs if large
require decision.md before promotion
```

---

## promoted

Run accepted as reference evidence.

Examples:

```text
baseline benchmark
validated improvement
important regression reference
```

Policy:

```text
never delete automatically
keep audit pack
keep config snapshot
keep analytics bundle
```

---

## rejected

Run has been reviewed and discarded.

Examples:

```text
overfit variant
unstable across slices
bad drawdown profile
guard relaxation without edge proof
```

Policy:

```text
delete after 14 days
keep decision summary only if useful
```

---

## archived

Run reduced to compressed evidence.

Examples:

```text
old candidate
historical comparison
not active but still informative
```

Policy:

```text
keep compressed audit pack
remove raw duplicate data
```

---

## protected

Special manual lock.

Examples:

```text
mentor packs
client evidence
production incident
legal/audit reference
```

Policy:

```text
never delete
never mutate
manual unlock required
```

---

# 7. Mandatory Run Artifact Contract

A run is only valid as research evidence if it contains:

```text
manifest.json
status.json
config_snapshot.yaml
metrics_summary.json
results_summary.md
decision.md
```

Optional but recommended:

```text
resolved_config.yaml
quantlog_events.jsonl
analytics/
audit_pack.zip
events_sample.jsonl
```

If mandatory files are missing, the run cannot be promoted.

---

# 8. Data Reduction Rules

## 8.1 Full QuantLog events

Full event logs are valuable but can become large.

Default policy:

```text
scratch: delete full events after 7 days
candidate: compress full events after 30 days
promoted: keep compressed full events or audit pack
rejected: delete full events after 14 days
```

---

## 8.2 Event sampling

For non-promoted runs, keep:

```text
first 200 events
last 200 events
all trade_action events
all risk_guard_decision events
all order lifecycle events
all trade_closed events
all anomaly events
```

Output:

```text
events_sample.jsonl
```

---

## 8.3 Analytics bundle

For every meaningful run, preserve:

```text
analytics/guard_attribution.json
analytics/funnel_summary.json
analytics/key_findings.md
analytics/EDGE_REPORT.md
analytics/PROMOTION_DECISION.md
```

If these do not exist, preserve whatever analytics output was generated and mark the run as incomplete.

---

# 9. Retention Policy

| Status    | Raw events | Analytics | Config | Decision | Action |
| --------- | ---------: | --------: | -----: | -------: | ------ |
| scratch   |     7 days |    7 days | optional | optional | delete |
| rejected  |    14 days |   14 days | keep if useful | keep summary | delete/reduce |
| candidate | 30–90 days |      keep | keep | required | compress |
| promoted  |    forever |   forever | forever | forever | protect |
| archived  | compressed |   reduced | keep | keep | protect |
| protected |    forever |   forever | forever | forever | protect |

# 10. Cleanup Script Design

Script location:

```text
quantmetrics_os/scripts/data_lifecycle.py
```

The script must operate only on:

```text
quantmetrics_os/runs/
quantanalytics/output_rapport/
quantbuild/logs/
quantbuild/data/quantlog_events/
```

The script must never modify:

```text
quantbuild/data/market_cache/
docs/*audit_pack*
docs/*context_pack*
promoted runs
protected runs
```

# 11. CLI Commands

## Dry run

```bash
python quantmetrics_os/scripts/data_lifecycle.py --dry-run
```

Shows what would be deleted, archived, compressed, or protected.

---

## Apply full policy

```bash
python quantmetrics_os/scripts/data_lifecycle.py --apply-policy
```

Applies lifecycle rules.

This command must refuse to run unless a dry-run report exists from the same day.

---

## Clean scratch runs

```bash
python quantmetrics_os/scripts/data_lifecycle.py --cleanup-scratch --days 7
```

---

## Clean rejected runs

```bash
python quantmetrics_os/scripts/data_lifecycle.py --cleanup-rejected --days 14
```

---

## Archive candidates

```bash
python quantmetrics_os/scripts/data_lifecycle.py --archive-candidates --days 30
```

---

## Protect promoted runs

```bash
python quantmetrics_os/scripts/data_lifecycle.py --protect-promoted
```

---

## Generate lifecycle report

```bash
python quantmetrics_os/scripts/data_lifecycle.py --report
```

Outputs:

```text
quantmetrics_os/reports/data_lifecycle_report.md
quantmetrics_os/reports/data_lifecycle_report.json
```

# 12. Safety Rules

The script must enforce:

```text
1. dry-run by default
2. no deletion of promoted runs
3. no deletion of protected runs
4. no deletion of market_cache
5. no deletion of docs mentor packs
6. no deletion if status.json is missing, unless explicitly marked as orphan
7. no deletion of config_snapshot.yaml
8. no deletion of decision.md
9. no deletion of audit_pack.zip
```

If a folder has no `status.json`, classify it as:

```text
unknown
```

Unknown runs are not deleted automatically.

---

# 13. Unknown Run Handling

If run folder lacks `status.json`, create:

```json
{
  "status": "unknown",
  "delete_allowed": false,
  "reason": "missing status.json; manual classification required"
}
```

Unknown runs require manual review.

Command:

```bash
python quantmetrics_os/scripts/data_lifecycle.py --classify-unknown
```

---

# 14. Archive Pack Format

When archiving a candidate, create:

```text
archive_pack.zip
```

Contents:

```text
manifest.json
status.json
config_snapshot.yaml
resolved_config.yaml
metrics_summary.json
results_summary.md
decision.md
events_sample.jsonl
analytics/
```

Optional:

```text
quantlog_events.jsonl.gz
```

After successful archive creation, raw duplicate files may be removed if:

```text
archive integrity check passes
status != promoted
status != protected
```

---

# 15. Integrity Checks

Before deletion or compression:

```text
- confirm file exists
- confirm archive can be opened
- confirm mandatory files are inside archive
- write lifecycle_action_log.jsonl
- write before/after file size summary
```

Lifecycle actions must be logged to:

```text
quantmetrics_os/logs/data_lifecycle_actions.jsonl
```

Each action:

```json
{
  "timestamp_utc": "...",
  "action": "archive_candidate",
  "path": "...",
  "run_id": "...",
  "status": "candidate",
  "bytes_before": 123456,
  "bytes_after": 45678,
  "dry_run": false
}
```

# 16. Promotion Requirements

A run can only move to `promoted` if it has:

```text
config_snapshot.yaml
metrics_summary.json
results_summary.md
decision.md
analytics evidence
```

Promotion should also require a human decision:

```md
# decision.md

## Verdict

PROMOTE / REJECT / HOLD

## Reason

...

## Evidence

...

## Known Weaknesses

...

## Next Test

...
```

---

# 17. What Must Never Happen

The lifecycle system must never:

```text
delete market data cache
delete promoted runs
delete protected runs
delete audit packs
delete context packs
delete configs required for reproduction
delete decision.md
treat loose analytics output as canonical
promote a run based only on PnL
```

---

# 18. First Implementation Scope

Do not build the full system at once.

Build V0 first:

```text
1. scan quantmetrics_os/runs/
2. detect run folders
3. read status.json if present
4. classify missing status as unknown
5. print dry-run table
6. calculate folder sizes
7. write data_lifecycle_report.md
```

No deletion in V0.

---

# 19. V1 Implementation Scope

After V0 is trusted:

```text
1. delete scratch runs older than 7 days
2. delete rejected runs older than 14 days
3. compress candidate raw events older than 30 days
4. protect promoted/protected runs
5. log every lifecycle action
```

---

# 20. Final Rule

QuantMetrics does not store everything.

QuantMetrics stores:

> The minimum evidence required to reproduce, explain, reject, or promote a decision system.

Raw data is temporary.

Evidence is permanent.
